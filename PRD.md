# PRD — 개빠름 메모검색 (세컨드 브레인 빠른 검색기)

## 1. 한 줄 정의
여러 폴더에 흩어진 텍스트 문서(txt/md/log 등)를 대상으로, 키워드를 입력하면 Everything급 속도로 파일과 내용을 찾아주는 macOS용 CLI 검색 도구.

## 2. 배경 및 문제 정의
- 세컨드 브레인(개인 지식관리, RAG 구성) 워크플로우를 운영하면 시간이 지날수록 다수의 폴더에 txt/md/log 문서가 누적된다.
- macOS Spotlight는 범용 검색이라 인덱싱 범위/속도/필터링을 세밀하게 통제하기 어렵고, `grep`/`ripgrep`은 매번 전체 스캔이라 문서 수가 많아질수록 느려진다.
- Windows의 [Everything](https://www.voidtools.com/)은 NTFS MFT를 직접 읽고 USN Journal로 실시간 갱신하여 "거의 즉시" 파일명 검색을 제공한다. 이 프로젝트는 그 체감 속도를 "파일명"이 아니라 "파일 내용" 검색에 적용하는 것이 목표다. (상세 분석: [BENCHMARK_EVERYTHING.md](BENCHMARK_EVERYTHING.md))

## 3. 목표 (Goals)
- G1. 사용자가 지정한 여러 폴더 트리 안의 txt/md/log 문서를 대상으로 키워드 입력 시 체감 즉시(수십~수백 ms) 검색 결과를 반환한다.
- G2. 문서가 추가/수정/삭제/이동되면 전체 재색인 없이 인덱스가 실시간으로 갱신된다.
- G3. 키워드(전문 검색, full-text) 검색을 1차 기본 기능으로 제공하고, 동일 인덱스/CLI 위에서 시맨틱(임베딩) 검색을 옵션 모드로 제공한다.
- G4. 검색 결과는 파일 경로 + 매칭된 줄/문맥 스니펫 + 관련도 순위를 함께 보여준다.

## 4. 비목표 (Non-goals, 이번 단계)
- Windows/Linux 동시 지원 (1차는 macOS 전용, 추후 확장 가능하도록만 설계)
- GUI/데스크톱 앱, 웹 UI (1차는 CLI)
- PDF/이미지/오피스 문서 OCR 등 비텍스트 포맷 파싱
- 클라우드 동기화, 멀티 디바이스 인덱스 공유
- LLM 기반 RAG 답변 생성(질의응답 챗봇) — 시맨틱 검색은 "관련 문서/구절 찾기"까지만, 답변 생성은 범위 밖

## 5. 타겟 사용자
- 본인(혹은 비슷한 워크플로우를 가진 1인 사용자): Obsidian/노션 내보내기, 회의록, 로그, 메모 등을 폴더별로 누적 관리하는 사람.
- CLI/터미널 사용에 익숙함. macOS 사용.

## 6. 핵심 기능 요구사항 (Functional Requirements)
| ID | 요구사항 | 설명 |
|---|---|---|
| F1 | 폴더 등록/관리 | 검색 대상 폴더(들)를 등록/제외 패턴(.gitignore 스타일) 설정 |
| F2 | 초기 색인 (Full Index Build) | 등록된 폴더를 스캔하여 txt/md/log 파일 내용을 전문 인덱스로 구축 |
| F3 | 실시간 증분 색인 | 파일 생성/수정/삭제/이동을 감지해 인덱스를 즉시 갱신 (전체 재스캔 없이) |
| F4 | 키워드 검색 | 키워드/구문 입력 → 매칭 파일 목록 + 스니펫 + 관련도 순 정렬, 결과 스트리밍 출력 |
| F5 | 검색 연산자 | AND/OR/NOT, 따옴표 구문검색(`""`), 와일드카드, 확장자/경로 필터 |
| F6 | 시맨틱 검색 모드 | `--semantic` 옵션 시 쿼리를 임베딩하여 의미상 유사한 문서/구절 검색 |
| F7 | 결과 출력 옵션 | plain/json 출력, 결과 개수 제한(`--limit`), 컨텍스트 줄 수(`-C`) 조절 |
| F8 | 인덱스 상태 확인 | 색인 파일 수, 마지막 갱신 시각, 인덱스 크기 등 상태 명령(`status`) |

## 7. 비기능 요구사항 (NFR)
- **속도**: 인덱스가 구축된 상태에서 키워드 검색은 체감 즉시(목표: p50 < 50ms, p95 < 200ms, 1만 파일/수백MB 텍스트 기준). **Phase 0 스파이크 실측(1만 파일/11.9MB, macOS 로컬): 평균 0.07ms** — 목표 대비 압도적으로 여유 있음. 다만 합성 코퍼스/단순 키워드 기준이므로 Phase 1에서 실제 사용 패턴(긴 구문, 복합 연산자)으로 재검증 필요. 상세: [BENCHMARK_EVERYTHING.md](BENCHMARK_EVERYTHING.md) 5장.
- **증분 갱신 지연**: 파일 변경 후 인덱스 반영까지 1초 이내(목표). **Phase 0 스파이크 실측: FSEvents 감지 지연 평균 11.7ms / 최대 14.3ms (200건 연속, 누락 0건)** — 목표를 두 자릿수 배수로 상회.
- **자원 사용**: 상시 실행되는 백그라운드 프로세스는 유휴 시 CPU 0%대, 메모리는 색인 대상 규모에 비례하되 인덱싱 대상 파일 1만 개 기준 수십 MB 수준을 목표.
- **신뢰성**: 비정상 종료 후에도 인덱스가 손상되지 않고 재시작 시 일관 상태로 복구. **✅ Phase 5 보강 (2026-06-19)**: 색인 대상 개별 파일이 스캔/색인 도중 사라지거나(레이스 컨디션) 읽기 권한이 없어져도 해당 파일만 건너뛰고 나머지 파일 색인은 계속 진행(이전에는 예외가 전파되어 전체 색인 루프가 중단됨). 상세: [PLAN.md](PLAN.md) Phase 5.
- **이식성(설계 차원)**: 1차는 macOS(FSEvents) 전용이 목표였으나, `watchdog` 라이브러리가 OS별 백엔드를 추상화해 Linux(inotify)도 별도 작업 없이 이미 동작 — CI macOS/Linux 매트릭스에서 매 커밋 검증됨(Phase 4 확인). Windows(USN Journal)는 watchdog가 지원하되 미검증 상태로 범위 밖 유지.

## 8. 성공 지표 (Success Metrics)
- M1. 색인된 코퍼스(예: 1만 개 파일, 300MB)에서 키워드 검색 응답 시간이 `ripgrep` 전체 스캔보다 명확히 빠름(목표: 5배 이상). **✅ Phase 0 실측: 약 2,650배 (1만 파일/11.9MB 합성 코퍼스 기준)**.
- M2. 파일 1개 수정 후 1초 이내에 검색 결과에 변경 내용이 반영됨. **✅ Phase 0 실측: FSEvents 평균 11.7ms로 감지. Phase 2(`sbsearch watch`)에서 전체 파이프라인(이벤트 감지 → 디바운스 → 재색인 → 검색 반영) 구현 및 수동 스모크 테스트로 1초 이내 반영 확인 — 정식 레이턴시 벤치마크(평균/최대치 실측)는 미실시, 필요 시 Phase 4에서 추가**.
- M3. 시맨틱 검색이 키워드가 정확히 일치하지 않는 질문(예: 동의어/요약 표현)에 대해서도 관련 문서를 상위 5위 안에 반환. **✅ Phase 3 검증**: 한국어 비키워드 질의(예: "오늘 회의에서 예산안을 논의했다")로 키워드가 겹치지 않는 관련 문서("이번 분기 예산 계획을 검토했다")를 무관 문서보다 상위에 반환함을 단위테스트(`tests/test_embeddings.py`)와 CLI 스모크 테스트로 확인. 대규모 코퍼스 기준 정량 벤치마크(정확도/순위 지표)는 미실시 — 필요 시 Phase 4에서 추가.

## 9. 제약사항/가정
- 대상 파일은 일반 텍스트(UTF-8 가정, 추후 인코딩 자동 감지 고려)이며 개인 사용 규모(수천~수십만 파일)를 가정. Everything처럼 수백만 파일 규모는 1차 목표가 아님.
- 시맨틱 검색은 프라이버시(개인 메모)를 고려해 외부 API가 아닌 로컬 임베딩 모델 사용을 기본 전제로 한다(구체 모델은 PLAN.md에서 결정).
