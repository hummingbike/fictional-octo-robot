# Everything 벤치마크 분석 — 무엇을 빌려오고, 무엇을 다르게 할 것인가

[Everything](https://www.voidtools.com/)(voidtools, Windows)을 기준점으로 삼아 "왜 빠른지"를 분석하고, macOS에서 파일 내용(content) 검색에 어떻게 적용할지 정리한다.

## 1. Everything이 빠른 이유

| 메커니즘 | 설명 |
|---|---|
| MFT 직접 읽기 | 디렉터리를 일반 API(FindFirstFile 등)로 순회하지 않고, NTFS의 **Master File Table을 직접 읽어** 전체 볼륨의 파일/폴더 메타데이터를 한 번에 가져온다. 수백만 개 파일도 초 단위로 초기 색인 완료. |
| USN Change Journal | NTFS의 변경 기록(USN Journal)을 구독해 생성/삭제/이름변경/수정 이벤트를 실시간으로 받는다. **전체 재스캔 없이** 인덱스를 증분 갱신. |
| 초경량 메모리 인덱스 | 파일명만 다루는 압축된 자료구조(정렬된 리스트/트라이 유사)를 메모리에 유지 — 100만 파일당 약 1MB 수준으로 알려져 있음. 디스크 I/O 없이 즉시 필터링 가능. |
| 검색 대상이 "이름"뿐 | 기본 검색은 **파일명/경로 문자열**만 매칭한다. 내용(content)을 보는 게 아니라서 매 키 입력마다 메모리 내 문자열 매칭만 하면 된다 — 그래서 "즉시" 체감이 가능. |
| Content Search는 별도, 느림 | Everything에도 내용 검색 옵션이 있지만 이는 **색인되어 있지 않고, 매칭된 파일을 그때그때 열어서 스캔**하는 방식이라 파일명 검색보다 훨씬 느리다. |

**핵심 시사점**: Everything의 "즉시성"은 "파일명만 다루는 사전 구축 인덱스 + 저비용 실시간 갱신" 조합에서 나온다. 우리가 만들 "내용 검색"은 본질적으로 더 무거운 작업(텍스트를 읽고 토큰화)이므로, Everything과 똑같은 속도를 내용 검색에서 그대로 재현하는 것은 목표가 아니라 — **같은 설계 원칙(사전 색인 + 증분 갱신, 매 쿼리마다 재스캔 금지)을 내용 검색에 적용**하는 것이 목표다.

## 2. macOS에는 MFT/USN Journal이 없다 — 대안

| Windows (Everything) | macOS 대응 | 비고 |
|---|---|---|
| MFT 직접 읽기 (초기 전체 스캔) | 병렬 디렉터리 워크 (`fts()`/Swift `FileManager` enumerator, 멀티스레드) 또는 기존 macOS 인덱스(Spotlight `mdfind`)를 보조 활용 | macOS는 MFT 같은 단일 구조가 없어 직접 읽기가 불가. 병렬 워크로 충분히 빠른 초기 색인 가능 (개인 규모 기준) |
| USN Change Journal (실시간 변경 구독) | **FSEvents API** | macOS의 표준 실시간 파일시스템 변경 알림 메커니즘. USN Journal과 동일한 역할(전체 재스캔 없는 증분 갱신)을 한다 |
| 이름만 색인 | 내용까지 색인 (전문검색, FTS) | 우리 프로젝트의 핵심 차별점. Apple Spotlight(`mds`)도 사실 콘텐츠까지 인덱싱하는 점에서 구조적으로 더 가까운 참고 모델 |

즉, **FSEvents = 우리의 USN Journal**, **사전 구축된 전문 인덱스(FTS) = 우리의 MFT**로 매핑해서 설계한다.

## 3. 우리 프로젝트가 다르게 가야 하는 지점

1. **검색 대상이 내용이라 색인 비용이 크다.** 파일명 문자열 비교가 아니라 텍스트를 읽고 토큰화해 역색인(inverted index)을 만들어야 한다. → 초기 색인은 "초 단위"가 아니라 "코퍼스 크기에 비례한 합리적 시간"을 목표로 한다 (개인 규모 수만 파일 기준 수십 초~분 단위 허용, 이후 증분 갱신만 빠르면 됨).
2. **갱신은 파일 단위로 국소화 가능.** 파일 하나가 바뀌면 그 파일의 색인 항목만 다시 만들면 되므로, FSEvents 이벤트 1건 → 해당 파일 재색인 1건으로 비용을 상수화할 수 있다 (Everything의 USN Journal 처리와 동일한 설계 원칙).
3. **랭킹이 필요하다.** 파일명 매칭은 "있다/없다"면 충분하지만, 내용 검색은 관련도 순위(BM25 등)가 사용자 경험에 직결된다. Everything에는 없는 요구사항.
4. **시맨틱 검색은 Everything에 대응점이 없는 완전히 추가된 레이어.** 별도 임베딩 인덱스(벡터)로 분리해서 설계한다.

## 4. 참고 측정 기준 (우리 프로젝트의 "Everything다움" 정의)
Everything의 절대 응답시간(<1ms급, 파일명 한정)을 그대로 목표로 삼지 않는다. 대신:
- **무거운 작업(텍스트 읽기/토큰화)은 색인 시점에 끝내고, 검색 시점에는 인덱스 조회만 한다** — 이 원칙을 지키는지가 "Everything답게 설계했는가"의 기준.
- 검색 시점 쿼리는 디스크 풀스캔이 한 번도 발생하지 않아야 한다(매번 모든 파일을 열어보는 방식은 실패 기준).
- 구체적 수치 목표는 [PRD.md](PRD.md) 8장(성공 지표) 참조.
